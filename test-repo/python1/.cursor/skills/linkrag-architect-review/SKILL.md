# LinkRAG 架构师视角 Review

以**系统架构师**的角色，对项目做**代码检查工具之外**的校验：目录与命名、分层与依赖、复杂度与可维护性、与既有架构文档的一致性。**不空洞**：先由脚本产出指标，AI 再根据指标 + 记忆 + 策略产出报告；支持记忆与策略配置，使审核**不呆板、可延续、随团队需求调换重点**。

## 执行流程（指标先行 + 记忆 + ReAct）

1. **指标**：若存在 `reports/metrics-backend.json`、`reports/metrics-frontend.json`，**必须先读取**并作为报告依据；若不存在，则提示用户先执行 `.\scripts\collect_metrics.ps1` 再继续。报告中的目录符合性、超长文件、router/views 列表等应**直接引用指标中的数据**，避免空洞描述。
2. **记忆**：读取 `.cursor/review-memory.md`（历史结论、团队偏好）。若存在「待跟进」或「上期问题」，本次报告中应**对照检查是否已改进**并注明。
3. **策略**：读取 `docs/review-strategy.md`。若其中填写了「本期优先」「放宽项」等，**据此调整**本轮 Review 的权重与结论侧重点；未配置则按下列默认维度执行。
4. **ReAct**：
   - **Observe**：从指标中观察 `unexpected_top_level`、`files_over_400_lines`、`dir_structure_ok`、`api_paths` 等；从记忆中观察上期问题与团队偏好。
   - **Reason**：结合策略决定本期重点（如本期只强调目录与复杂度，或重点看前后端接口对应）；判断哪些指标异常需要进一步抽样代码说明。
   - **Act**：对需要说明的项做必要代码抽样（如某超长文件、某意外目录）；然后撰写报告。
5. **报告**：按下方「输出格式」产出报告，结论**引用具体指标字段与文件路径**。
6. **更新记忆**：报告完成后，将本次 2～3 条关键发现或决策**追加**到 `.cursor/review-memory.md` 的「历史结论摘要」中，并注明待跟进项（若有）。

## 使用场景

- 用户说：「从架构师角度 review 一下项目」「检查目录结构和命名规范」「做一次系统层面的校验」「复杂度、耦合度有没有问题」
- 在重要迭代或重构前，做一次架构健康度检查

## 参考依据

- **架构与目录约定**：`docs/开发框架说明.md`（backend/frontend 结构、路由与响应约定、联调要点）
- **可扩展性分析**：`docs/前后端可扩展性分析.md`（扩展点、瓶颈、改进建议）
- **规约**：`.cursor/rules/` 下与 backend/frontend 相关的规则
- **代码检查**：`docs/代码检查.md` 与 `scripts/check_*.ps1` 已覆盖的（ESLint/Prettier/Stylelint/ruff/py_compile）不再重复，本 Review 关注**上述工具不覆盖**的层面
- **质量模型参考**：Review 维度与 **SonarQube 质量模型**（Reliability、Security、Maintainability、Coverage、Duplications）对齐，详见 `docs/SonarQube与检查标准.md`

## Review 维度与检查项

### 1. 目录结构与命名规范

- **后端**：是否严格符合 `src/backend` 约定（`main.py`、`config.py`、`database/`、`routers/`、`llm/`、`logger/`、`server/`）；新增目录/文件是否与现有命名风格一致（小写、短横线或下划线）；是否存在游离在约定外的目录或重复职责的目录。
- **前端**：是否符合 `src/frontend/src` 下 `api/`、`router/`、`stores/`、`views/`、`layouts/`、`components/`、`constants/`、`styles/`、`utils/` 等划分；`api/modules` 是否按业务模块拆分、命名是否与后端路由语义对应；`views` 下是否按功能/路由划分、命名是否清晰。
- **命名一致性**：后端路由前缀、前端 API 路径与文档是否一致；组件/文件命名是否统一（PascalCase 组件、kebab-case 或 camelCase 文件等，与项目既有习惯一致）。

### 2. 分层与依赖

- **后端**：路由层是否只做参数校验与调用 crud/服务，业务逻辑是否下沉到 database/crud 或 server；是否存在 router 直接写复杂 SQL 或跨模块强耦合；llm、server 与 routers 的依赖方向是否合理（如 main 注入 llm 到 router，而非 router 直接 import 全局单例以外的深层实现）。
- **前端**：views 是否通过 `api/modules` 访问后端、是否绕过统一 request 实例；store 与 api 的调用关系是否清晰；是否存在循环依赖或跨层直接访问（如 view 直接改另一个 view 的局部状态）。
- **跨端**：前后端接口契约（路径、请求/响应体）是否与 `docs/API文档.md` 或 `docs/接口契约.md` 一致；是否有未文档化的接口或字段。

### 3. 复杂度与可维护性

- **文件与函数**：是否存在单文件过长（如 >400 行）、单函数过长或圈复杂度过高；建议标注「建议拆分」的文件或函数，并简要说明理由。
- **重复与抽象**：是否存在明显重复逻辑（可抽取为公共函数、组件或公共 router 工具）；是否存在可复用的逻辑仍未抽象。
- **可测试性**：关键路径是否便于单测或集成测试（如依赖是否可注入、是否便于 mock）；是否有难以测试的全局状态或副作用集中点。

### 4. 规范与文档一致性

- **与开发框架说明**：当前目录、命名、路由与响应约定是否与 `docs/开发框架说明.md` 一致；若有偏差，列出并建议对齐方式。
- **与可扩展性分析**：若文档中已指出扩展点或技术债，当前代码是否与之相符或已改善；是否有新的技术债出现。
- **规约**：`.cursor/rules` 中的约定是否在本次 Review 范围内被遵守（可抽样检查）。

### 5. 其他架构关注点（可选）

- **安全**：敏感接口是否均有鉴权；敏感配置是否未硬编码。
- **性能与资源**：是否存在明显的 N+1 查询、大列表未分页、前端未做防抖/节流等可预见的性能问题。
- **可观测性**：关键操作是否有日志或可追踪性（本项可简要提及，不替代运维文档）。

## 执行注意

- **指标缺失时**：若未找到 `reports/metrics-*.json`，必须先提示用户运行 `.\scripts\collect_metrics.ps1`，再基于新生成的指标写报告；不得仅凭「印象」泛泛而谈。
- 若用户指定了范围（如「只 review 后端」「只 review 知识库模块」），则仅对指定范围执行上述维度，并在报告开头说明范围。
- 结论尽量具体到**指标中的 path/数据**与**文件/目录/接口名**，便于落地修改；建议可引用 `docs/开发框架说明.md` 或 `docs/前后端可扩展性分析.md` 的对应段落。
- 本 Skill 与「代码检查」（ESLint/Prettier 等）互补：代码检查侧重语法与格式，本 Review 侧重结构、命名、分层与架构一致性，且**以脚本指标为据**。
