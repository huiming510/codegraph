# LinkRAG 业务专家视角 Review（需求与实现对照）

以**业务/产品专家**角色，基于**需求文档**对当前**实现**做对照检查，识别已实现、部分实现、未实现与偏差，并给出可验收性建议。支持**记忆**与**策略**：历史结论与团队偏好写入 `.cursor/review-memory.md`，策略权重在 `docs/review-strategy.md` 中配置，使审核**不呆板、可延续、随团队需求调换重点**（如本期只对某模块或 P0 做对照）。

## 执行流程（记忆 + 策略 + ReAct）

1. **记忆**：读取 `.cursor/review-memory.md`。若存在「待跟进」或上期业务结论，本次应对照检查是否已关闭或改进，并在报告中注明。
2. **策略**：读取 `docs/review-strategy.md` 的「业务 Review 策略」。若填写了「本期优先」「需求文档路径」等，**据此调整**本轮重点与需求文档来源；未配置则使用默认 `docs/需求文档.md` 与全部维度。
3. **指标（可选）**：若存在 `reports/metrics-frontend.json`，可引用其中 `api_paths`、`views_list` 辅助「接口与依赖一致性」「功能覆盖」的对照，使结论有据可查。
4. **ReAct**：**Observe** 需求文档中的功能点与验收标准、记忆中的上期问题、策略中的本期优先；**Reason** 确定本轮重点（如只做 P0、或只做某模块）；**Act** 在代码中定位实现并填写覆盖表，再撰写报告。
5. **报告**：按下方「输出格式」产出，结论具体到需求编号与文件/接口/页面。
6. **更新记忆**：将本次 2～3 条关键结论或待跟进项**追加**到 `.cursor/review-memory.md` 的「历史结论摘要」中。

## 使用场景

- 用户说：「对照需求文档 check 一下实现」「从业务角度 review 需求和代码」「验收前做一次需求覆盖检查」「需求文档和实现是否一致」
- 迭代结束或提测前，做需求与实现的一致性检查

## 输入

- **需求文档**：默认使用 `docs/需求文档.md`；若用户指定其他文档，以用户指定为准。
- **实现范围**：默认覆盖 `src/backend` 与 `src/frontend` 中与需求相关的部分；若用户指定模块，仅对照该模块。

## 参考依据

- **业务实现检查标准**：`docs/业务实现检查标准.md` — 定义功能覆盖、验收达成、偏差与遗漏、可追溯性、可验收性等**衡量标准**，本 Review 的结论与输出格式应与之对齐。
- **需求文档结构**：文档信息、需求概述（背景/目标/范围）、功能需求（功能点、说明、优先级、验收标准）、非功能需求、接口与依赖、验收与发布。
- **实现**：后端 `routers/`、`database/crud`、`server/`；前端 `views/`、`api/modules/`、路由与菜单。
- **接口契约**：`docs/API文档.md`、`docs/接口契约.md`（若有）。

## Review 维度

### 1. 需求文档理解与提取

从需求文档提取：在范围内的功能/模块、各功能点（编号、说明、优先级）、验收标准、非功能需求、依赖接口。若文档为模板或未填写，说明后仅对已明确项做对照。

### 2. 功能覆盖度

对每个功能点在代码中定位实现（后端 router/接口、前端 view/api），标注：**已实现** / **部分实现** / **未实现** / **无法确定**，并简要说明依据。

### 3. 验收标准达成

对需求中的验收标准逐条判断是否满足、是否可测；若无验收标准，可建议补充 1～2 条合理验收条。

### 4. 偏差与遗漏

- **偏差**：实现与需求描述不一致。
- **遗漏**：需求在范围内但无对应实现。
- **多余**：实现中有需求未提及的功能（可提示）。

### 5. 边界与异常（业务视角）

需求或实现中是否考虑错误处理、空数据、权限等；非功能需求（性能、安全）是否有对应体现。

### 6. 接口与依赖一致性

需求中提到的 API 是否存在于实现与 API 文档；前后端接口路径与格式是否一致。

### 7. 代码注释与可维护性（防止垃圾代码堆叠）

依据 **`.cursor/rules/code-comments.mdc`**：需求涉及的**对外 API**（后端路由、前端 api/modules 导出方法）与**关键业务逻辑**是否有清晰注释（用途、参数/返回值或意图）。无注释或仅无信息量注释的，在报告中标注「需补充注释」或「建议补充注释」；存在大段废代码注释或注释与实现不符的，标注为不符合注释规范。

## 输出格式

```markdown
## LinkRAG 需求与实现对照报告

### 1. 需求范围摘要
（依据文档、在范围内项、未明确项）

### 2. 功能覆盖
（表：编号、功能点、优先级、实现情况、说明）

### 3. 验收标准达成
（已满足、未满足、建议补充）

### 4. 偏差与遗漏
（偏差、遗漏、其他）

### 5. 边界与异常
（已考虑、建议补充）

### 6. 接口与依赖
（一致、不一致或缺失）

### 7. 代码注释与可维护性
（对外 API 与关键逻辑注释情况、需/建议补充项；依据 code-comments.mdc）

### 8. 总结与建议
（可验收性、验收前建议完成项）
```

## 执行注意

- 需求文档为空或仅模板时，先提示补充需求，再对已填写部分做对照。
- 结论具体到需求编号与代码位置（文件、接口、页面），便于产品与开发对齐；可引用 `reports/metrics-frontend.json` 中的 api_paths/views_list 作为实现侧依据。
- 与「架构师 Review」互补：本 Skill 侧重需求与实现的对应与可验收性；架构 Review 侧重指标（目录、复杂度等），本处侧重功能点与验收标准。
