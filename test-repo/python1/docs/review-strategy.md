# Review 策略与偏好

本文件供**架构师 Review**与**业务 Review**技能使用。团队可在此声明本阶段/本迭代的 Review 重点，AI 会据此**调整权重与策略**，使专家审核不呆板、与时俱进、更贴合开发团队需求。

---

## 1. 架构 Review 策略

| 维度         | 默认权重 | 本期权重/说明 |
|--------------|----------|-------------------------------|
| 目录与命名   | 高       | （可填：高/中/低 或 本期略过） |
| 分层与依赖   | 高       |                                |
| 复杂度       | 中       |                                |
| 文档一致性   | 中       |                                |
| 安全/性能    | 低       |                                |

- **本期优先**：（可选）如「本迭代只做目录与命名 + 超 400 行文件」或「重点看前后端接口一致性」。
- **放宽项**：（可选）如「server/ 目录暂不要求与开发框架说明完全一致」。

---

## 2. 业务 Review 策略

业务实现的**衡量标准**见 **`docs/业务实现检查标准.md`**（功能覆盖、验收达成、偏差与遗漏、可追溯性、可验收性等）。下表仅配置各维度权重与本期重点。

| 维度           | 默认权重 | 本期权重/说明 |
|----------------|----------|-------------------------------|
| 功能覆盖       | 高       |                                |
| 验收标准达成   | 高       |                                |
| 偏差与遗漏     | 高       |                                |
| 边界与异常     | 中       |                                |
| 接口与依赖一致 | 中       |                                |

- **本期优先**：（可选）如「本迭代只对知识库与文档模块做需求对照」或「重点看 P0 验收标准」。
- **需求文档**：默认 `docs/需求文档.md`；若本迭代使用其他文档，在此注明路径。

---

## 3. 通用约定

- **记忆**：每次 Review 后，AI 会将关键结论与待跟进项追加到 `.cursor/review-memory.md`，下次 Review 会参考并检查「上次问题是否已改进」。
- **指标先行**：架构 Review 前应先运行 `.\scripts\collect_metrics.ps1`（或 `.\scripts\review_prepare.ps1`），AI 基于 `reports/metrics-backend.json` 与 `reports/metrics-frontend.json` 出报告，避免空洞描述。
- **策略切换**：修改本文件或 `.cursor/review-memory.md` 中的「团队偏好」即可调换下一轮 Review 的重点，无需改技能代码。
- **充分性评估**：从架构师视角对当前检查与 Review 的覆盖度、缺口与优先级建议，见 **`docs/架构师视角-检查与Review充分性评估.md`**。
