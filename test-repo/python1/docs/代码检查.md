# 代码检查说明

项目提供统一脚本用于**后端**与**前端**的代码检查，可在提交前或需要时执行。

---

## 1. 一键检查（推荐）

在**项目根目录**执行：

```powershell
.\scripts\check_all.ps1
```

会依次执行后端检查、前端检查，并输出汇总结果。任一失败则退出码为 1。

**报告输出**：检查完成后会在项目根下生成 **`reports/`** 目录（已加入 .gitignore）：
- **`reports/check-report.md`**：汇总报告（生成时间、后端/前端状态、失败时最近输出、处理建议）。仅在执行 **`check_all.ps1`** 时生成。
- **`reports/check-backend.log`**：后端检查的完整输出（由 `check_backend.ps1` 写入）。
- **`reports/check-frontend.log`**：前端检查的完整输出（由 `check_frontend.ps1` 写入）。

让 AI 解读报告并给出处理方式：使用项目技能 **read-check-report**（`.cursor/skills/read-check-report/`）。对 AI 说「看下检查报告」「如何处理检查结果」等，AI 会读取 `reports/check-report.md` 与对应 log，并列出问题与修复建议。

---

## 2. 分别检查

### 2.1 后端（src/backend）

```powershell
.\scripts\check_backend.ps1
```

- **语法检查**：使用 `python -m py_compile` 对 `src/backend` 下所有 `.py` 做语法检查（不依赖额外安装）。
- **ruff 检查**（可选）：若已安装 [ruff](https://docs.astral.sh/ruff/)，会额外执行：
  - `ruff check src/backend`：规范检查（风格、潜在问题）
  - `ruff format --check src/backend`：格式检查（是否需格式化）

**安装 ruff 以启用规范与格式检查：**

```powershell
pip install ruff
```

**后端格式自动修复：**

```powershell
ruff format src/backend
ruff check src/backend --fix
```

ruff 配置见项目根目录 `pyproject.toml` 中 `[tool.ruff]`。

### 2.2 前端（src/frontend）

```powershell
.\scripts\check_frontend.ps1
```

内部执行 `npm run check`，即：

- **ESLint**：`eslint .`
- **Stylelint**：样式检查（仅检查，不自动修复）
- **Prettier**：`prettier --check` 检查格式

**在前端目录下直接执行：**

```powershell
cd src\frontend
npm run check        # 仅检查
npm run check:fix    # 检查并自动修复（ESLint --fix、Stylelint --fix、Prettier --write）
```

## 3. 脚本与命令对照

| 目的           | 命令 / 脚本 |
|----------------|-------------|
| 后端 + 前端一起检查 | `.\scripts\check_all.ps1` |
| **一键自动修复**（格式与可自动修复的规范） | `.\scripts\check_fix.ps1`，建议修复后再跑 `check_all.ps1` |
| **提交前全量门禁**（检查 + 测试） | `.\scripts\run_full_check.ps1` |
| **Review 准备**（采集指标 + 生成清单） | `.\scripts\review_prepare.ps1`，再让 AI 执行架构/业务 Review 技能 |
| 依赖漏洞扫描（P1） | `.\scripts\check_audit.ps1`，报告 `reports/check-audit.log`；可选 `-Strict` 按漏洞失败 |
| 仅后端检查     | `.\scripts\check_backend.ps1` |
| 仅前端检查     | `.\scripts\check_frontend.ps1` 或 `cd src/frontend && npm run check` |
| 前端检查并修复 | `cd src/frontend && npm run check:fix` |
| 后端格式修复   | `ruff format src/backend`、`ruff check src/backend --fix` |

脚本总览见 **`scripts/README.md`**。

---

## 4. 使用场景

- **提交前**：运行 `.\scripts\check_all.ps1` 确保无语法/规范/格式问题。
- **CI**：在流水线中执行 `check_all.ps1`（或分别执行 backend + frontend 脚本），失败则禁止合并。
- **仅改前端时**：在 `src/frontend` 下执行 `npm run check` 或 `npm run check:fix`。
- **仅改后端时**：执行 `.\scripts\check_backend.ps1`，若需格式化再执行 `ruff format src/backend`。
- **查看/解读报告**：打开 `reports/check-report.md` 查看汇总；或让 AI 根据「检查报告」「如何处理检查结果」读取报告并给出修复步骤（依赖 read-check-report 技能）。

## 5. 专家 Review 与指标采集

代码检查（ESLint/Prettier/ruff 等）侧重语法与格式。**架构师 Review** 与 **业务 Review** 则依赖**脚本产出的指标**，再由 AI 根据指标 + 记忆 + 策略出报告，避免空洞、并可随团队需求调换重点。

### 5.1 采集指标与 Review 准备（脚本先行）

在项目根目录执行以下任一方式：

- **方式一（推荐）**：`.\scripts\review_prepare.ps1` — 先采集指标，再生成 `reports/review-checklist.md`，按清单提示使用 AI Review 技能。
- **方式二**：`.\scripts\collect_metrics.ps1` — 仅采集指标。

会生成（均在 `reports/` 下，已 gitignore）：

- **`reports/metrics-backend.json`**：后端目录结构、文件行数、超 400 行文件列表、router 文件列表等。
- **`reports/metrics-frontend.json`**：前端目录结构、文件行数、api 模块与请求路径、views 列表等。
- **`reports/metrics-summary.md`**：简要说明，供 AI 知晓指标文件用途。

### 5.2 架构师 Review

- 使用技能 **linkrag-architect-review**。AI 会**先读取**上述 JSON 指标与 `.cursor/review-memory.md`、`docs/review-strategy.md`，再按 ReAct（观察指标 → 结合策略确定重点 → 抽样代码 → 出报告）产出架构 Review 报告。
- 若未运行过 `collect_metrics.ps1`，AI 会提示先执行再继续。

### 5.3 业务 Review（需求与实现对照）

- 使用技能 **linkrag-business-review**。AI 会读取需求文档（默认 `docs/需求文档.md`）、记忆与策略，并可引用 `reports/metrics-frontend.json` 中的 api_paths/views_list 辅助对照，产出需求覆盖与验收建议。
- 策略与记忆使审核不呆板、可延续：可在 `docs/review-strategy.md` 中填写「本期优先」维度，在 `.cursor/review-memory.md` 中积累历史结论与团队偏好。

## 6. SonarQube：参考标准或直接使用

- **参考 SonarQube 的检查标准**：不跑 SonarQube，仅在架构 Review 与文档中按 SonarQube 质量模型（可靠性、安全、可维护性、覆盖率、重复）对齐检查维度。详见 **`docs/SonarQube与检查标准.md`**。
- **直接使用 SonarQube**：配置 `sonar-project.properties` 中的 `sonar.host.url`、`sonar.login` 后，在项目根目录执行 `.\scripts\run_sonar.ps1`（需已安装 SonarScanner CLI），在 SonarQube/SonarCloud 查看质量门禁与报告。与现有 `check_all.ps1`、`collect_metrics.ps1` 可并行使用。

**业务实现检查标准**：衡量「需求是否被正确、完整实现」的标准（功能覆盖、验收达成、偏差与遗漏、可追溯性等）见 **`docs/业务实现检查标准.md`**；业务 Review 技能（linkrag-business-review）按该文档执行对照，策略与权重在 `docs/review-strategy.md` 中配置。
