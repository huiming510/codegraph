# 测试策略与用例生成

本文约定**测试分层**（单元 / 疏通 / 集成）、**用例自动生成策略**与 **TDD 模式**，以及对应的文档与追溯方式。

---

## 一、测试分层定义

| 层级 | 名称 | 目标 | 范围 | 依赖 | 运行频率 |
|------|------|------|------|------|----------|
| **单元测试** | Unit | 单函数/单类/单组件行为正确 | 最小可测单元，mock 外部 | 无 DB/真实 API | 每次保存/提交 |
| **疏通测试** | Smoke | 主路径可跑通、无致命错误 | 接口 200、页面可挂载 | 测试库/内存 DB，mock 外部服务 | MR 前、CI 主分支 |
| **集成测试** | Integration | 多模块协作、关键流程端到端 | 跨路由/前后端/DB 或 ES | 测试库或专用 test 环境 | 合并前、发布前 |

### 1.1 单元测试（Unit）

- **后端**：单函数、单路由处理逻辑、工具函数；mock `get_db`、外部 HTTP、LLM 等。
- **前端**：单组件、单 store action、工具函数；mock `@/api`、router、pinia 等。
- **特点**：快、稳定、不依赖网络与真实 DB，适合 TDD 与频繁运行。
- **放置**：后端 `src/backend/tests/unit/` 或与实现同包 `tests/unit/`；前端 `src/**/__tests__/unit/*.spec.js` 或按模块 `*.spec.js`。

### 1.2 疏通测试（Smoke）

- **后端**：关键接口 HTTP 200、响应体结构符合约定（如 `code==200`）；可带真实测试库（如 `test_db.db`）。
- **前端**：关键页面/组件能挂载、关键文案或元素存在；可 mock API 返回。
- **特点**：覆盖主路径，保证「能跑通」，是 CI 门禁与 MR 前必跑项。
- **放置**：后端 `src/backend/tests/test_*.py`（当前约定）；前端 `src/**/__tests__/*.spec.js` 或 `tests/smoke/*.spec.js`。

### 1.3 集成测试（Integration）

- **后端**：多路由协作、真实 SQLite/ES（或 test 配置）、解析/RAG 流程中关键步骤。
- **前端**：多页面流程、真实 API（对本地后端）或 contract 测试。
- **特点**：更接近真实环境，运行较慢，适合发布前或定时跑。
- **放置**：后端 `src/backend/tests/integration/`；前端可选 `tests/integration/` 或 E2E 仓库。

## 二、用例自动生成策略

### 2.1 触发时机

| 时机 | 建议生成内容 | 执行方 |
|------|----------------|--------|
| 开发完成某模块后 | 该模块的疏通用例 + 核心路径的单元用例 | 开发 / AI（linkrag-test-cases） |
| MR 前 | 补齐 MR 改动范围的疏通用例 | 开发 / AI |
| 需求/验收确定后（TDD） | 先写失败用例，再实现 | 开发 / AI |
| 发布前 | 集成用例或 E2E 关键路径 | 开发 / 脚本 |

### 2.2 按层级生成规则

- **单元**
  - **依据**：函数/类/组件签名、入参类型、简单等价类（正常/边界/异常）。
  - **生成**：每个导出函数/组件至少 1 个「正常」用例；边界与异常按复杂度追加。
  - **AI 提示**：「为 `routers/xxx.py` 的 `create_item` 生成单元测试，mock DB 与外部调用。」
- **疏通**
  - **依据**：路由表、接口路径、前端路由与主要 view。
  - **生成**：每个路由模块至少 1～2 个「主路径 200」；每个主要页面至少 1 个「挂载 + 关键元素存在」。
  - **AI 提示**：「为知识库模块生成疏通测试：列表 200、创建 200、带 token 的删除 200。」
- **集成**
  - **依据**：用户故事、需求文档中的「验收步骤」、关键业务流程。
  - **生成**：每条关键流程 1 个用例（如：登录 → 创建知识库 → 上传文档 → 问答一次）。
  - **AI 提示**：「按需求文档『上传并解析文档』验收，生成一条集成测试。」

### 2.3 与需求/接口的对应（可追溯）

- **需求 → 用例**：在需求文档或 TODO 中为每条验收赋予 ID（如 `REQ-001`）；生成用例时在测试描述或 docstring 中注明 `REQ-001`，便于追溯。
- **接口 → 用例**：后端按 `routers/` 模块分文件（如 `test_knowledge_bases.py`），与 API 文档/Swagger 对应；前端按 view 或 api 模块分目录。
- **文档**：在 `docs/需求文档.md` 或迭代文档中保留「验收项 ↔ 测试文件/用例名」的简单表格（可选）。

## 三、TDD 模式（测试驱动开发）

### 3.1 流程（Red → Green → Refactor）

1. **Red**：根据需求/验收先写**失败**的测试（预期行为未实现）。
2. **Green**：写最少实现让测试通过。
3. **Refactor**：在测试保持通过的前提下重构实现与测试代码。

### 3.2 与文档的对应（可执行规格）

- **需求/验收文档**：写清「给定…当…那么…」或检查清单，可作为用例描述的直接来源。
- **测试用例**：把上述描述变成可执行的 test（函数名/描述即规格），实现「文档即可执行」。
- **顺序建议**：先定验收（需求文档或 TODO）→ 再写测试（单元或疏通）→ 再写实现；MR 时附「需求/验收 ↔ 用例」说明。

### 3.3 何时用 TDD

- **新功能、新接口、新组件**：优先先写测试再实现，便于设计接口与边界。
- **修 Bug**：先写一个「暴露该 Bug」的失败用例，修完再通过。
- **遗留代码**：先补疏通用例保证主路径，再按需补单元与集成，不必一步到位。

## 四、用 Skills 生成用例与辅助材料

### 4.1 是否能用 Skills 做测试 case

可以。项目已提供 **linkrag-test-cases** Skill（`.cursor/skills/linkrag-test-cases`），在用户说「给 xxx 写测试」「疏通测试」「单元测试」「TDD」等时，由 AI 按该技能生成对应层级的用例，并放入约定目录（见 [测试体系](./测试体系.md)）。生成质量依赖**辅助材料是否齐全**。

### 4.2 辅助材料清单（生成前建议准备）

| 材料 | 用途 | 哪里找 / 怎么准备 |
|------|------|-------------------|
| **API 文档（含响应约定）** | **接口测试的唯一依据**：code、data、msg 等断言必须以文档为准 | `docs/API文档.md` 1.2 节；成功时 `code==0`，勿用 `code==200` |
| **需求/验收文档** | 集成用例、TDD 用例、用例描述与需求 ID 追溯 | `docs/需求文档.md`、迭代文档；每条验收建议有 ID（如 REQ-001） |
| **接口/路由清单** | 疏通用例：哪些接口要 200、路径与请求体 | `docs/API文档.md`、Swagger、`src/backend/routers/` 与 `main.py` 挂载 |
| **前端路由与主要 view** | 疏通用例：哪些页面要挂载、关键元素 | `src/frontend/src/router/`、`views/` 与菜单配置 |
| **带注释的代码** | 单元/疏通：函数意图、参数与返回值、边界与异常 | 见下文「五、文档先行与代码注释」；遵守 `.cursor/rules/code-comments.mdc` |
| **现有 conftest / fixture** | 生成的用例与现有风格、client/auth 一致 | `src/backend/tests/conftest.py`、前端 test 工具或 mock 约定 |

**根本原则**：**文档不全则用例易错**。接口测试的断言（如 `code`、`data` 结构）必须严格依据 `docs/API文档.md`；文档未定义或模糊时，生成用例会出错。先补全文档，再生成或修改用例。

**最小可用**：仅有路由/接口列表 + 主要 view 列表，即可生成疏通用例；若再提供需求验收与注释良好的代码，可生成单元与集成用例并带需求追溯。

### 4.3 使用 Skill 时的提示建议

- 指定范围与层级：「为 `routers/knowledge_bases.py` 生成疏通测试」「为 `views/chat/list` 生成挂载 + 关键元素用例」。
- 提供依据：「按 `docs/需求文档.md` 中 REQ-002 的验收步骤生成一条集成测试」。
- TDD：「按 TDD 为登录接口先写测试，再实现」；此时应先有验收描述或接口约定。

## 五、文档先行与代码注释（测试依据）

### 5.1 生成代码的同时是否要记录文档？测试才有依据

**要。** 建议约定：

- **先有依据再测**：需求/验收（文档）或接口契约（API 文档/Swagger）先确定，再写测试或生成用例；这样用例才有「预期行为」可依。
- **与代码同步记录**：开发某功能时，在需求文档或迭代文档中写下该功能的验收项（可执行的一条条「给定…当…那么…」）；实现完成后，同一批验收可作为集成/疏通用例的描述来源；MR 时附「需求/验收 ↔ 测试文件」便于追溯。
- **文档即可执行规格**：验收写清楚后，测试用例就是把验收变成可执行的 test；这样「文档 → 测试 → 实现」或「文档 + 实现 → 测试」都有一致依据。

未写文档时也可生成疏通用例（仅保证 200/挂载），但**单元与集成**、以及**需求追溯**会缺少依据，故建议至少对关键功能保留简短验收或接口说明。

### 5.2 开发代码是否需要有足够的注释？

**必须有。** 项目已通过 [.cursor/rules/code-comments.mdc](../.cursor/rules/code-comments.mdc) 和 [业务实现检查标准](./业务实现检查标准.md) 约定：

- **必须注释**：文件/模块职责、**对外 API（路由、api 模块导出）**的用途与主要参数/返回值、复杂业务逻辑与非显而易见的分支/常量。
- **注释是测试生成的重要输入**：AI 根据函数/路由的 docstring 或 JSDoc 理解「做什么、参数与返回、边界」，才能生成有针对性的单元用例（正常/边界/异常）和更准确的疏通断言；无注释时只能做泛泛的「调通 200」或简单挂载。
- **与业务 check 一致**：注释不足在业务实现检查中会标注为「需补充」；注释与实现不一致则必须修正。

因此：**开发代码需要有足够注释**，既满足规约与可维护性，也便于用 Skills 生成高质量测试 case。

## 六、分层与 CI / 门禁的对应

| 测试层级 | 个人分支 | main/develop MR | 发布/release |
|----------|----------|------------------|--------------|
| 单元 | 可选（本地跑） | 可选 | 建议跑 |
| 疏通 | 不强制 | **必过** | **必过** |
| 集成 | 不跑 | 可选 | 建议跑 |

当前 GitLab CI（见 [GitLab分支与质量门禁](./GitLab分支与质量门禁.md)）主分支门禁为「Check + 疏通测试」；若后续加入单元/集成，可在同一 pipeline 中增加对应 job，并按分支规则控制是否必过。

## 七、AI 生成用例时的约定（与 linkrag-test-cases 对齐）

- **疏通优先**：用户未明确说「单元」或「集成」时，默认生成疏通用例（接口 200、页面挂载）。
- **单元**：用户说「单元测试」「为 xxx 函数写 test」时，生成单函数/单组件级用例，并 mock 外部依赖。
- **集成**：用户说「集成测试」「按验收写一条流程测试」时，生成跨模块/带真实依赖的用例，并注明所需环境（如测试库、ES）。
- **TDD**：用户说「按 TDD」「先写测试」时，先根据需求/接口写出失败用例，再提示用户实现对应逻辑。

用例生成后应放在本文与 [测试体系](./测试体系.md) 约定的目录中，并保持与现有 conftest、fixture 一致。
