# 业务实现检查标准

本文档定义**业务实现**层面的检查标准，用于衡量「需求是否被正确、完整地实现」。与代码规范（ESLint/ruff）、架构 Review 互补：此处关注**功能覆盖、验收达成、偏差与遗漏**等业务可验收性。

业务 Review 技能（**linkrag-business-review**）按此标准执行需求与实现对照；策略与权重在 `docs/review-strategy.md` 中配置。

---

## 1. 衡量维度与标准

| 维度 | 标准（达标即通过） | 检查方式 |
|------|---------------------|----------|
| **需求可追溯** | 需求文档中在范围内的每条功能/验收标准，能在代码中定位到对应实现（后端接口/前端页面或接口调用） | 逐条功能点 → 标注实现位置（文件、路由、view） |
| **功能覆盖** | 范围内功能均有明确实现状态：**已实现** / **部分实现** / **未实现** / **无法确定**，且依据可说明 | 功能覆盖表：编号、实现情况、依据（接口/页面/代码路径） |
| **验收标准达成** | 需求中写明的验收标准逐条可判断「已满足 / 未满足 / 需补充」；未写验收标准的可建议补 1～2 条可测条款 | 验收标准表：条款、是否满足、是否可测 |
| **偏差可控** | 实现与需求描述一致；若不一致（偏差），需在报告中列出并标注是否已评审/接受 | 偏差列表：需求描述 vs 实际实现、建议处理 |
| **范围内无遗漏** | 需求「在范围内」的项均有对应实现，无遗漏；若有遗漏须在报告中列出并建议补全或调整范围 | 遗漏列表：需求编号、功能点、建议 |
| **边界与异常（业务视角）** | 错误处理、空数据、权限、超时等有考虑或文档说明；非功能需求（性能、安全）有对应体现 | 边界/异常与非功能项检查表 |
| **接口与依赖一致** | 需求中提到的 API 在实现与 API 文档中存在；前后端路径、请求/响应格式一致 | 需求提及接口 ↔ 实现/文档对照 |
| **代码注释与可维护性** | 对外 API（后端路由、前端 api/modules 导出）与复杂业务逻辑有清晰注释（用途、参数/返回值或意图）；无大段废代码注释、无与实现不符的注释 | 依据 `.cursor/rules/code-comments.mdc`；对无注释的对外接口与关键逻辑标注「需补充注释」 |

---

## 2. 代码注释规则（防止垃圾代码堆叠）

业务实现检查时，可依据**代码注释**判断可维护性：对外接口与关键业务逻辑无注释或注释无效，易导致“垃圾代码”堆叠，不利于需求对照与后续维护。

- **规约位置**：`.cursor/rules/code-comments.mdc`（适用于 backend + frontend）。
- **必须注释**：文件/模块头、对外 API（路由、api/modules 导出方法）、复杂业务逻辑、非显而易见的分支或常量；注释需说明「做什么」及必要时「为什么」，参数/返回值需在对外接口处写明。
- **禁止**：大段注释掉的废代码（应删除）、无信息量注释（如“这里循环”）、与实现不一致的注释。
- **业务 check 用法**：对需求涉及的后端路由与前端接口方法、关键逻辑做抽样；缺用途说明或参数/返回值说明的，在报告中列为「需补充注释」；复杂逻辑无意图说明的，列为「建议补充注释」。

---

## 3. 与需求文档的对应关系

- **需求文档**（默认 `docs/需求文档.md`）应包含：**功能点**（编号、说明、优先级）、**验收标准**、**在范围内/范围外**、**接口与依赖**。
- 业务实现检查时：
  - 以「在范围内」的功能与验收标准为**必查项**；
  - 「不在范围内」的仅标注不检查；
  - 若文档为空或仅模板，先提示补充需求，再对已填写部分按上表标准检查。

---

## 4. 结论与可验收性

- **通过**：范围内需求可追溯、功能覆盖完整、验收标准已满足或可测、无未处理偏差与遗漏、边界与接口一致。
- **有条件通过**：存在少量偏差或遗漏，但已列出且团队同意延后或调整范围；报告中注明「验收前建议完成项」。
- **不通过**：存在范围内遗漏、验收标准不可测或大量未满足、关键偏差未记录或未评审；报告中明确列出并建议先修复再验收。

---

## 5. 与其它检查的关系

| 检查类型 | 关注点 | 标准/工具 |
|----------|--------|-----------|
| 代码规范与格式 | 语法、风格、格式 | ESLint、Prettier、ruff、`check_*.ps1` |
| 架构与可维护性 | 目录、分层、复杂度、与文档一致 | 架构师 Review、metrics、可参考 SonarQube 质量模型 |
| **业务实现** | 需求覆盖、验收、偏差与遗漏、注释与可维护性 | 本文档 + 需求文档 + linkrag-business-review + code-comments.mdc |

三者可组合使用：先跑代码检查与指标，再做架构 Review，最后做业务实现对照，作为提测/验收前的标准流程。
