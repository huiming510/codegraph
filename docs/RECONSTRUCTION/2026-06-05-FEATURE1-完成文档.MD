## 二、实现细节

### 2.1 schema.py — Pydantic 模型

共定义了以下数据模型：

```
RuleVersion (str enum)
ConditionType (str enum)  ← ast_query / xpath / regex / annotation / naming / relation
BindingSource (str enum) ← attr / text / node

MatchTarget         ← 单个匹配目标的过滤条件
BindingSpec         ← 从 AST 查询中提取命名 capture 的结果
Condition           ← 单一匹配条件
EmitEdge            ← 从匹配结果生成边
Rule               ← 规则文件完整模型（顶层）
RuleBundle         ← 规则包元数据
```

**关键设计决策：**1111

- `Rule.version` 字段固定为 `"1"`，用于标识规则格式版本，与语义化版本 `rule_version` 区分
- `RuleBundle.description` 类型为 `str | dict`，支持单语言字符串或多语言 `zh/en` 字典
- `extra = "forbid"` 严格禁止未定义字段，防止用户输入 typo 静默忽略
- `Rule.match` 使用 `dict[str, MatchTarget]` 而非固定 `source`/`target` 字段，保留扩展性

**校验函数：**

- `validate_rule(yaml_content: str | dict | Path) -> (Rule, errors)` — 接受字符串、字典或文件路径，返回校验结果和精确错误位置
- `validate_bundle(bundle_dir: Path) -> (bool, errors)` — 校验规则包完整性（metadata 存在、rules 目录存在、所有规则文件存在且通过 Schema 校验）

### 2.2 v1/rule_ast.py — Rule AST 运行时结构

将 Pydantic `Rule` 模型转换为运行时结构：

```
RuleAST            ← 规则完整 AST
MatchTargetAST     ← MatchTarget 编译版本（regex 已预编译）
ConditionAST       ← Condition 编译版本
EmitEdgeAST        ← EmitEdge 编译版本
BindingPathSpec    ← 绑定路径解析（text / attr=X / key=X / parent.X）
RuleASTBuilder     ← 从 Rule 构建 RuleAST 的构建器
```

**BindingPathSpec** 解析 `key=property`、`attr=class`、`parent.attr` 等路径格式，提取 `path_type` 和 `path_value`，支持多层 `parent` 遍历。

### 2.3 v1/operators.py — 操作符与变换

- `OperatorRegistry` — 按条件类型索引的操作符注册表，支持 `ast_query`、`regex`、`naming`、`annotation`、`xpath`、`relation`
- `apply_transform(value, transform_name)` — 支持 `camel_to_snake`、`snake_to_camel`、`trim`
- `parse_ref_template(template)` — 解析 `target:{class}.{property}` 等引用模板，提取前缀类型和绑定变量名
- `render_ref_template(template, bindings)` — 用实际绑定值替换模板占位符

### 2.4 内置规则包

#### java-orm

- `metadata.yaml` — 定义 id=`java-orm`，目标语言 java，框架 mybatis/spring-data-jpa
- `rules/mybatis-orm.yaml` — 3 种条件类型（annotation、ast_query、ast_query target=source），1 条 sql_mapping 边

#### web-framework

- `metadata.yaml` — 定义 id=`web-framework`，目标语言 java/xml/jsp，框架 servlet/struts/jsp
- `rules/jsp-java.yaml` — regex/ast_query 条件，jsp_java_form / jsp_java_usebean / jsp_java_render 边
- `rules/webxml-servlet.yaml` — xpath/annotation 条件，web_xml_to_servlet / web_xml_to_filter 边

#### examples

- `mybatis-orm.yaml` — 完整 mybatis 规则示例，含注释
- `jsp-java.yaml` — JSP-Java 映射示例，含注释
- `webxml-servlet.yaml` — WebXML-Servlet 映射示例，含注释

## 四、使用方式

### 4.1 校验规则文件

```python
from scripts.plugins.rules_engine.schema import validate_rule

rule, errors = validate_rule("path/to/rule.yaml")
if errors:
    for e in errors:
        print(e)  # e.g. "  [conditions[0].type] Invalid condition type..."
```

### 4.2 校验规则包

```python
from pathlib import Path
from scripts.plugins.rules_engine.schema import validate_bundle

bundle_dir = Path("scripts/plugins/rules_engine/bundles/built-in/java-orm")
valid, errors = validate_bundle(bundle_dir)
assert valid, "\n".join(errors)
```

### 4.3 构建 Rule AST

```python
from scripts.plugins.rules_engine.schema import validate_rule
from scripts.plugins.rules_engine.v1.rule_ast import RuleASTBuilder

rule, _ = validate_rule("path/to/rule.yaml")
ast = RuleASTBuilder(rule).build()

print(ast.source.language)     # "java"
print(ast.binding_names)       # {"property", "column", ...}
print("property" in ast.from_ref_bindings)  # True
```

### 4.4 模板解析

```python
from scripts.plugins.rules_engine.v1.operators import parse_ref_template, render_ref_template

prefix, base, bindings = parse_ref_template("target:{class}.{property}")
# prefix="target", base="{class}.{property}", bindings=["class", "property"]

result = render_ref_template("target:{class}.{property}", {"class": "User", "property": "id"})
# result = "target:User.id"
```

## 五、与其他 FEATURE 的关系

FEATURE1 定义了整个规则引擎的数据结构和格式基础，后续阶段依赖这些定义：

| 阶段 | 依赖 FEATURE1 的内容 |
|------|---------------------|
| FEATURE2 | 复用 `Rule`、`RuleBundle`、`validate_rule/validate_bundle`；`RuleASTBuilder` 进一步扩展 |
| FEATURE3 | 复用 `ConditionAST`、`BindingPathSpec` 进行匹配执行 |
| FEATURE4 | 复用 `EmitEdgeAST` 和 `parse_ref_template/render_ref_template` 生成边 |
| FEATURE5 | 复用 `RuleAST` 和 `RuleASTBuilder` 进行规则加载与执行 |
| FEATURE6 | 复用 `RuleBundle` 和 `validate_bundle` 进行规则包管理 |

## 六、已知限制与后续改进

1. **REPL 标签缺失** — `RuleAST` 中 `BindingPathSpec` 对 `REPL` 标签（如 `@Result(property="userId")` 中提取 `property` 属性值）需要后续在 FEATURE3 的条件执行器中实现解析逻辑
2. **Tree-sitter 查询预编译** — `RuleASTBuilder` 目前仅验证 regex pattern，Tree-sitter 查询的预编译将在 FEATURE2 中实现（需要传入 language 参数）
3. **条件组合逻辑（AND/OR/NOT）** — 当前 Schema 支持 `combinator` 和 `sub_conditions`，但实际求值逻辑在 FEATURE3 实现
4. **依赖规则加载顺序** — `depends_on` 字段已定义，但循环依赖检测在 FEATURE5 实现
