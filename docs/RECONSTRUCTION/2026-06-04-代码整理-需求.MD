# CodeGraph Scripts 项目整理需求

> 整理日期：2026-06-04
> 整理范围：`scripts/` 目录

---

## 一、背景与目标

当前 `scripts` 目录在多次迭代中遗留了大量历史版本的冗余结构、命名和接口，代码膨胀且语义混乱。主要问题集中在以下几个方面：

1. **CLI 接口混乱** — 多种入口模式并存，只有 `--all` 和 `--singlefile` 真正有用
2. **Pipeline 层级概念不清** — stage0~stage6 并非同一层次的串行流水线，旧版5-stage已废弃
3. **插件目录冗余** — 多个带数字前缀的子目录（frontend/backend/embedded/tpl/cfg/build），需精简为 language/otherfiles/filecross 三个
4. **输出目录三级嵌套** — 子图按 layer/subcat/type 三级分类，改为按类型后缀平铺
5. **Handler 接口不统一** — SourceCodeHandler 与其他 Handler 签名不一致

本文档系统梳理所有待整理项，作为后续重构的执行依据。

## 二、CLI 接口整理

### 2.1 当前状态

`run_pipeline.py` 的 `argparse` 支持四种入口模式：

| 模式 | 参数 | 实际功能 | 状态 |
|------|------|----------|------|
| `--all` | `run_all()` | 全量分析，所有语言 + 跨文件边 | **保留** |
| `--singlefile` | `run_singlefile()` | 按类型单文件子图 + 相关跨文件边 | **保留** |
| `--lang <lang>` | `run()` | 单语言旧版5阶段流程 | **废弃 → 删除** |
| 无参数 + 无 `--all` | `run()` 自动检测 | 降级为 `--lang java` | **废弃 → 删除** |

此外还有：
- `--auto`：自动发现 `src` 和 `out` 目录
- `--mode filtered|full`：图输出模式

### 2.2 待整理项

- [ ] **删除废弃入口**：移除 `--lang` 单语言模式和 `run()` 函数（包含旧版5阶段流程）
- [ ] **删除自动降级逻辑**：移除无参数时的 `run()` 自动检测和降级行为
- [ ] **保留 `--all`**：作为默认/全量分析入口
- [ ] **保留 `--singlefile`**：作为按类型单文件分析入口
- [ ] **CLI 帮助文档清理**：移除 `--lang` 描述，更新 `--singlefile` 说明为"按文件类型解析并输出其子图及相关的跨文件边"

## 三、Pipeline 流程概念整理

### 3.1 当前状态

代码中存在**两套并行**的 Pipeline 概念：

```
旧版 5-stage Pipeline（stage1_syntax → stage5_semantic）：
  stage1: 语法解析（AST）
  stage2: 符号表构建
  stage3: 类型推断
  stage4: 引用解析
  stage5: 语义增强

新版 --all Pipeline：
  Phase 0: 文件分类（stage0_classify.py）
  Phase 1: 子图构建（run_for_type() 统一 Handler）
  Phase 2: 子图合并
  Phase 3: 跨文件边生成（stage6_cross_file.py）
```

`run_pipeline.py` 中：
- `run()` = 旧版5-stage（单语言）→ **删除**
- `run_all()` = 新版全流程（多语言）
- `run_singlefile()` = 新版子流程

### 3.2 待整理项

- [ ] **删除旧版 Pipeline**：移除 `run()` 函数及 `stage1_syntax.py`、`stage2_symbols.py`、`stage3_types.py`、`stage4_refs.py`、`stage5_semantic.py` 的旧版调用链
- [ ] **统一 Pipeline 命名**：统一使用 `--all` = `run_all()`
- [ ] **重命名阶段编号**：
  - `stage0_classify.py` → `phase0_classify.py` 或 `classify.py`
  - `stage6_cross_file.py` → `phase1_cross_file.py` 或 `cross_file.py`
  - 原因：`stage0` 和 `stage6` 暗示它们是 `stage1~stage5` 的前置/后置阶段，但实际上它们与旧版5-stage 不在同一抽象层次
- [ ] **删除中间文件输出**：确认 `--all` 模式下不生成 `stage1_ast.json`、`stage2_symbols.json`、`stage3_typed.json`、`stage4_refs.json`、`stage5_graph.json` 等中间文件（`run_all()` 函数已确认不生成这些，仅输出子图 JSONL + 最终产物）

## 五、Handler 接口统一

### 5.1 当前状态

`run_pipeline.py` 定义了 5 个 Handler 类：

| Handler | `build_subgraph` 返回 | `write_jsonl` 参数 | 问题 |
|---------|----------------------|-------------------|------|
| `SourceCodeHandler` | `(dict, SubgraphStats)` | `(graph, ast_data, ui_data, ...)` | 独特签名 |
| `TemplateHandler` | `(list, list, SubgraphStats)` | `(symbols, edges, entries, ...)` | 三元组 |
| `ConfigHandler` | `(list, list, SubgraphStats)` | `(symbols, edges, entries, ...)` | 三元组 |
| `BuildHandler` | `(list, list, SubgraphStats)` | `(symbols, edges, entries, ...)` | 三元组 |
| `FrameworkHandler` | `(list, list, SubgraphStats)` | `(symbols, edges, entries, ...)` | 三元组 |

- `SourceCodeHandler` 使用 `dict` 格式，其余使用 `(symbols: list, edges: list)` 格式
- `SourceCodeHandler.write_jsonl` 接受 `(graph, ast_data, ui_data, out_dir, full_graph)`
- 其余 Handler 的 `write_jsonl` 接受 `(symbols, edges, entries, out_dir, full_graph)`
- `BuildHandler` 和 `FrameworkHandler` 的 `write_jsonl` 方法末尾定义 `ui_data = {}` 但从未使用

### 5.2 待整理项

- [ ] **统一 Handler 接口**：
  - `build_subgraph` 全部改为返回 `(symbols: list[dict], edges: list[dict], stats: SubgraphStats)`
  - `write_jsonl` 统一为 `(symbols, edges, entries, out_dir, full_graph)`
- [ ] **移除未使用变量**：`BuildHandler.write_jsonl` 和 `FrameworkHandler.write_jsonl` 末尾的 `ui_data = {}` 删除
- [ ] **合并重复的子图写入逻辑**：所有 Handler 的 `write_jsonl` 实现高度相似（写入 `nodes.jsonl` + `edges.jsonl`），考虑抽象为公共函数
