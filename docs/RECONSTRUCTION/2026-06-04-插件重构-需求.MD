# 插件重构需求：规则文件驱动的新一代跨文件分析

## 一、背景与目标

### 1.1 现状问题

当前 CodeGraph 的跨文件关系分析完全由 Python 代码实现，存在以下根本性缺陷：

| 问题 | 描述 | 影响 |
|------|------|------|
| **开发门槛高** | 编写跨文件关系需要精通 Python + Tree-sitter API + 图谱数据模型 | 业务人员无法参与 |
| **版本管理困难** | 关系规则和代码混杂，无法独立版本化 | 无法像依赖一样管理关系定义 |
| **难以社区分享** | 规则无法打包分发，无法在团队/社区间共享 | 无法建立生态 |
| **修改成本高** | 每次调整关系匹配逻辑都需改代码、跑 CI | 迭代周期长 |
| **难以调试** | 规则逻辑散落在多个 Python 文件中 | 排查问题困难 |
| **无法可视化预览** | 规则表达的是什么关系，无法直观看到 | 配置容易出错 |

### 1.2 核心目标

**让插件开发完全脱离 Python 代码**，改用结构化的**规则文件**来描述跨文件关系，实现：

1. **声明式规则**：用 YAML/JSON/DSL 描述"什么样的代码模式产生什么样的边"
2. **版本化管理**：规则文件可独立提交 PR、独立版本发布
3. **社区分享**：规则包可发布到仓库供他人复用
4. **即时生效**：修改规则无需修改 Python 代码，无需重新部署
5. **可视化编辑**：提供 UI 工具预览规则匹配结果

| **规则编辑器** | 可视化编辑和预览规则 | Web UI（独立工具） |

## 三、规则文件格式设计

### 3.1 核心概念

```
Rule Package (规则包)
├── metadata.yaml          # 包元数据
├── rules/                # 规则目录
│   ├── rule-id-1.yaml
│   ├── rule-id-2.yaml
│   └── ...
└── schemas/              # 可选：自定义类型定义
```

### 3.2 规则文件结构

```yaml
# 规则文件示例：MyBatis ORM 映射
id: mybatis-entity-mapping
name:
  zh: MyBatis 实体映射
  en: MyBatis Entity Mapping
description: |
  建立 MyBatis XML/注解 中的 resultMap / @Results
  与 Java Entity 之间的属性映射关系。

version: 1.0.0
author: community
tags: [java, orm, mybatis, cross-file]

# 匹配条件：满足以下条件的文件对才会被处理
match:
  # 源文件条件：MyBatis Mapper XML 或带 @Mapper 的 Java 接口
  source:
    - language: xml
      file_pattern: "**/*Mapper.xml"
    - language: java
      annotation: "@Mapper"

  # 目标文件条件：Java Entity 类
  target:
    language: java
    class_annotation: "@Entity"
    # 或通过方法签名推断：findBy* 方法的返回类型

# 匹配条件：具体的代码模式
conditions:
  # 模式1：XML resultMap 中的 result 节点
  - name: resultMap-property
    pattern:
      language: xml
      query: |
        //resultMap[@id=$mapId]/result[@property=$prop and @column=$col]
    bindings:
      mapId:   "@mapId"
      prop:    "@property"
      col:     "@column"

  # 模式2：Java Mapper 接口中的 @Results 注解
  - name: results-annotation
    pattern:
      language: java
      query: |
        (class_declaration
          (annotation
            (annotation
              name: (identifier) @ann-name
              (annotation_arguments
                (annotation
                  (annotation
                    name: (identifier) @results-ann
                    (annotation_arguments
                      (annotation
                        name: (identifier) @result-ann
                        (annotation_arguments
                          (element
                            (pair
                              key: (identifier) @key
                              value: (string) @value)))))))))
    # 使用 tree-sitter 查询

# 边生成规则
edges:
  - name: entity-field-mapping
    from: target              # 从 target（Entity）出发
    to: source                # 到 source（Mapper）
    kind: "sql-mapping"       # 边类型
    from_qname: "EntityClass.{prop}"
    to_qname: "MapperInterface.findBy*"
    properties:
      mapped_column: "{col}"
      mapping_type: "resultMap"
```

### 3.3 规则条件表达式

```yaml
conditions:
  # 条件1：基于文件路径
  - file_path: "**/mapper/**/*.xml"

  # 条件2：基于 AST 模式（tree-sitter query）
  - ast_pattern: "(method_declaration name: (identifier) @name)"

  # 条件3：基于注解
  - annotation: "@RestController"

  # 条件4：基于方法名模式
  - method_name_pattern: "findBy*|get*|load*"

  # 条件5：基于类型
  - return_type: "List<*>"

  # 条件6：组合条件（AND/OR/NOT）
  - and:
      - annotation: "@Service"
      - method_name_pattern: "*Service"

  # 条件7：基于上下文字段传播（核心能力）
  - context_propagate:
      from: result-column
      through: method-call
      to: database-field
```

### 3.4 字段传播规则（核心）

这是实现"从 HTTP 参数到数据库字段"完整调用链的关键机制：

```yaml
# 示例：Java 到 SQL 的字段传播
id: field-propagation-chain
name:
  zh: 字段传播链
  en: Field Propagation Chain

description: |
  追踪字段从 HTTP 层到数据库层的完整传播路径：
  HTTP参数 → Controller形参 → Service实参 → Repository方法
  → JPA派生方法 → JPQL → SQL WHERE子句 → DB列

match:
  source:
    language: java
  target:
    language: sql

edges:
  - name: parameter-propagation
    chain:
      # 步骤1：HTTP参数 → Controller参数
      - source: "HTTP request parameter"
        predicate: "param.name == handler.param.name"
        target: "method parameter"

      # 步骤2：Controller参数 → Service调用实参
      - source: "method parameter"
        predicate: "call.arg.name == param.name"
        target: "method call argument"

      # 步骤3：Service实参 → Repository方法
      - source: "call argument"
        predicate: "repo.method.param.name == arg.name"
        target: "repository method parameter"

      # 步骤4：Repository方法 → SQL列
      - source: "repository method"
        predicate: "method.name matches derived-query-pattern"
        target: "SQL column"
```

## 四、规则包（Rule Bundle）格式

### 4.1 包元数据

```yaml
# metadata.yaml
package:
  id: "com-codegraph-java-orm"
  name:
    zh: "Java ORM 关系规则包"
    en: "Java ORM Relationship Rules"
  version: "1.2.0"
  description: |
    包含 MyBatis、Hibernate、JPA、MyBatis-Plus 的
    跨文件关系定义。

  author: "CodeGraph Team"
  license: "MIT"
  homepage: "https://github.com/codegraph/rules-java-orm"

  # 依赖的其他规则包
  dependencies:
    - id: "com-codegraph-base"
      version: ">=1.0.0"

  # 适用的语言/框架
  targets:
    languages: [java]
    frameworks: [mybatis, hibernate, jpa]

  # 规则列表
  rules:
    - id: "mybatis-entity-mapping"
      file: "rules/mybatis-entity-mapping.yaml"
      version: "1.0.0"

    - id: "jpa-repository"
      file: "rules/jpa-repository.yaml"
      version: "1.1.0"
```

### 4.2 规则包版本管理

- **语义化版本**：主版本.次版本.修订号
- **依赖解析**：支持 `>=`, `^`, `~` 等版本约束
- **冲突检测**：同名规则不同版本时的告警策略
- **覆盖机制**：本地规则优先于远程规则

### 5.2 执行流程

```
输入：GraphContext (symbols, edges, ast_data, ...)
          │
          ▼
   ┌─────────────────┐
   │   加载规则包     │  ◀── 从配置文件/远程仓库加载
   └────────┬────────┘
            │
            ▼
   ┌─────────────────┐
   │   解析规则文件   │  ◀── YAML → Rule AST
   └────────┬────────┘
            │
            ▼
   ┌─────────────────┐
   │   校验规则      │  ◀── Schema + 语义检查
   └────────┬────────┘
            │
            ▼
   ┌─────────────────┐
   │   过滤匹配文件   │  ◀── 根据 match.source/target 条件
   └────────┬────────┘
            │
            ▼
   ┌─────────────────┐
   │   逐条执行规则   │
   └────────┬────────┘
            │
      ┌─────┴─────┐
      ▼           ▼
 ┌────────┐  ┌────────┐
 │条件匹配│  │链式传播│
 └────┬───┘  └────┬───┘
      │           │
      └─────┬─────┘
            ▼
   ┌─────────────────┐
   │   生成边        │
   └────────┬────────┘
            │
            ▼
   ┌─────────────────┐
   │   去重/排序     │
   └────────┬────────┘
            │
            ▼
输出：新增的 Edge 列表 → 合并到 GraphContext
```

### 5.3 规则执行模式

支持两种执行模式：

1. **逐文件模式**：对每个满足条件的文件执行规则
2. **跨文件模式**：对文件对（source → target）执行规则
3. **全遍历模式**：扫描所有可能的组合（谨慎使用）
4. **增量模式**：只对变更的文件重新执行

## 六、现有 Python 插件的迁移策略

### 6.1 迁移路径

| 现有插件 | 迁移方式 | 优先级 |
|---------|---------|--------|
| `jsp_java.py` | 重写为规则文件 | P1 |
| `mybatis_orm.py` | 重写为规则文件 | P1 |
| `web_xml_servlet.py` | 重写为规则文件 | P1 |
| `dicon_component.py` | 重写为规则文件 | P2 |
| `tld_handler.py` | 重写为规则文件 | P2 |
| `template_code.py` | 重写为规则文件 | P2 |

### 6.2 兼容性保障

- **双轨并行**：新规则引擎与旧 Python 插件并行运行
- **结果对比**：新旧结果自动对比，差异告警
- **回滚能力**：可通过配置切换回纯 Python 模式
- **渐进迁移**：按优先级逐个迁移，每迁移一个验证一个

## 七、CLI 交互设计

### 7.1 规则相关命令

```bash
# 规则包管理
codegraph rule install <package-id>[@version]    # 安装规则包
codegraph rule uninstall <package-id>            # 卸载规则包
codegraph rule list                             # 列出已安装规则
codegraph rule update [package-id]              # 更新规则包
codegraph rule search <keyword>                 # 搜索规则市场

# 规则开发
codegraph rule init <rule-id>                   # 初始化新规则
codegraph rule validate <rule-file>             # 校验规则文件
codegraph rule test <rule-file> --sample <dir>  # 用样本数据测试规则
codegraph rule preview <rule-file> --target <path>  # 预览规则匹配结果

# 规则执行
codegraph analyze <path> --with-rules <rule-id> # 使用指定规则分析
codegraph analyze <path> --no-builtin            # 禁用内置规则
codegraph analyze <path> --rule-config <yaml>   # 自定义规则配置
```

### 7.2 规则配置文件

```yaml
# .codegraph/rules.yaml
rules:
  enabled:
    - "mybatis-entity-mapping"
    - "jpa-repository"
    - "field-propagation-chain"

  disabled:
    - "legacy-template-binding"

  overrides:
    "mybatis-entity-mapping":
      match:
        source:
          file_pattern: "**/custom-mapper/**/*.xml"
```

## 八、非功能性需求

### 8.1 性能目标

| 指标 | 目标值 |
|------|--------|
| 规则加载时间 | < 500ms（含网络缓存） |
| 单条规则匹配延迟 | < 100ms / 文件 |
| 规则引擎内存占用 | < 现有 pipeline 的 20% 增量 |
| 增量分析（10%文件变更） | < 全量分析的 15% 时间 |

### 8.2 可观测性

- 规则执行日志：详细的匹配/未匹配原因
- 性能剖析：每条规则的执行耗时统计
- 覆盖率报告：多少符号/文件被规则覆盖

### 8.3 安全性

- 规则沙箱：限制规则的文件访问范围
- 签名验证：规则包来源校验
- 恶意规则检测：禁止危险操作（如网络请求、文件写入）

## 十、开放问题

1. **规则版本冲突**：当两个规则包定义了同一对文件间的边，如何裁决？
2. **循环依赖检测**：规则 A 依赖规则 B 的输出，规则 B 又依赖规则 A，如何处理？
3. **规则调试体验**：当规则匹配失败时，如何快速定位问题？
