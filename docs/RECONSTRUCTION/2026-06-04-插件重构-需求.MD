# 插件重构需求：规则文件驱动的跨文件分析引擎

> 整理日期：2026-06-04
> 关联设计：[2026-06-04-插件重构-设计.MD](./2026-06-04-插件重构-设计.MD)

---

## 一、背景与目标

### 1.1 现存问题

当前 CodeGraph 的跨文件（Cross-File）关系发现依赖 **Python 插件系统**（位于 `plugins/cross_file/plugins/`）。这一设计存在以下根本性缺陷：

| 问题 | 影响 |
|------|------|
| **代码编写门槛高** | 需要掌握 Python + Tree-sitter API + CodeGraph 图结构 |
| **跨文件关系难以版本化管理** | 插件与代码耦合，无法独立提交/分享/审查 |
| **无法社区化运营** | 框架/ORM/模板引擎的跨文件规则无法作为独立包发布 |
| **规则无法渐进迁移** | 每新增一种跨文件关系，都需要修改 Python 代码 |
| **调试困难** | 规则逻辑散布在数百行 Python 代码中，无可视化工具 |

### 1.2 核心目标

构建一套**规则文件驱动的跨文件分析引擎**：

> 插件开发完全脱离 Python 代码，用规则文件描述跨文件关系，可版本化管理、可社区分享。

### 1.3 核心价值

- **降门槛**：规则文件（YAML）描述跨文件关系，非 Python 开发者也能编写
- **可版本化**：规则与代码解耦，独立提交、Code Review、版本控制
- **可分享**：规则包（Rule Bundle）可通过 CLI 安装/卸载，形成生态
- **可验证**：规则执行结果与 Python 插件结果可对比，确保迁移正确性
- **可扩展**：新增框架/ORM 关系，只需编写 YAML 文件，无需修改 Python 代码

### 1.4 设计原则

1. **不破坏现有架构**：新规则引擎与 Python 插件双轨并行，共存于 stage6
2. **复用现有原子能力**：复用 stage1~5 的 Tree-sitter 解析、符号表、类型推断结果
3. **规则即数据**：YAML 规则文件经解析后直接驱动执行，无中间代码生成
4. **渐进迁移**：每迁移一个 Python 插件为一条规则，同步验证结果一致性

## 二、术语约定

| 术语 | 定义 |
|------|------|
| **规则文件** | 描述单一跨文件关系的 YAML 文件（`.yaml`） |
| **规则包** | 包含 `metadata.yaml` + 多个规则文件的目录 |
| **规则引擎** | 执行"规则文件加载→解析→匹配→边生成"全流程的 Python 模块 |
| **Rule AST** | 规则文件解析后的内存结构（dataclass 树） |
| **规则包管理器** | 通过 CLI 安装/卸载/管理规则包的模块 |
| **绑定（Binding）** | 从 AST 查询中提取的命名 capture 结果 |
| **匹配目标（MatchTarget）** | 规则中定义的 source/target 文件过滤条件 |

## 三、功能需求

### FEATURE 1：规则文件格式与 Schema 定义

**需求描述**：定义规则文件的 YAML 格式及 Pydantic Schema，确保所有规则文件格式一致、可校验。

#### F1.1 规则文件结构

规则文件必须包含以下顶层字段：

- `id`：唯一标识（snake_case），如 `mybatis-orm`
- `name`：多语言名称（`{zh: ..., en: ...}`）
- `description`：规则描述
- `version`：语义化版本（默认 `1.0.0`）
- `tags`：标签列表（用于分类和搜索）
- `match`：定义 source 和 target 两类匹配目标
- `conditions`：匹配条件列表
- `edges`：边发射规格列表
- `priority`：执行优先级（数字越小越先执行，默认 100）
- `enabled`：是否启用（默认 true）
- `depends_on`：依赖的其他规则 ID（高级特性）

#### F1.2 匹配目标（MatchTarget）

每个 `match` 包含 `source` 和 `target`，各自支持以下过滤条件：

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `language` | string | 目标语言 | `"java"` |
| `file_pattern` | string | glob 文件名模式 | `"**/*Mapper.xml"` |
| `annotation` | string | 注解名称（含 @） | `"@Mapper"` |
| `class_annotation` | string | 类级别注解 | `"@Entity"` |
| `method_pattern` | string | 方法名正则 | `"findBy\|get\|load"` |
| `return_type` | string | 返回类型模式 | `"List<*>"` |
| `kind` | string | 符号种类 | `"class"`, `"method"`, `"field"` |
| `paired_with` | MatchTarget | 对端文件约束（跨文件条件） | 见设计文档 |

#### F1.3 条件类型（Condition）

支持以下条件类型（ConditionType）：

| 类型 | 说明 | 查询方式 |
|------|------|----------|
| `ast_query` | Tree-sitter AST 查询 | tree-sitter query 语法 |
| `xpath` | XPath 查询 | XML 文件专用 |
| `regex` | 正则表达式匹配 | 内容匹配 |
| `annotation` | 注解匹配 | 符号表注解 |
| `naming` | 命名模式匹配 | 正则表达式 |
| `relation` | 符号间关系（跨文件） | 依赖其他规则结果 |

#### F1.4 绑定规格（Binding）

从 AST 查询结果中提取命名 capture 的值：

- `name`：capture 名称（binding 变量名）
- `source`：来源类型（`attr` / `text` / `node`）
- `transform`：可选变换（`camel_to_snake`、`snake_to_camel`、`trim`）

binding 路径格式支持：

- `"text"` — 节点文本内容
- `"attr=X"` — 属性值（XML）
- `"key=X"` — 注解中的 key=value 对
- `"parent.X"` — 父节点属性
- `"parent.parent.X"` — 多层父路径

#### F1.5 边发射规格（EmitEdge）

从匹配结果生成边，指定：

- `name`：边的标识名（用于日志/debug）
- `from_ref`：边起点引用（支持模板语法 `"{binding_name}"`）
- `to_ref`：边终点引用
- `kind`：边类型（如 `sql-mapping`、`renders`、`form-bound`）
- `properties`：边属性字典（支持模板语法）
- `line_ref`：行号来源的 binding 引用

引用模板支持格式：

- `"{binding_name}"` — 使用绑定变量
- `"target:{xpath}"` — 从 target 文件的 AST 中取值
- `"symbol:{qname}"` — 使用符号表中的 qname
- `"db::{column}"` — 字面量数据库引用
- `"meta:source"` / `"meta:target"` — 使用 match 中的元信息

#### F1.6 规则包元数据（RuleBundle）

规则包目录包含 `metadata.yaml`，定义：

- `id`：规则包标识
- `name`：多语言名称
- `version`：语义化版本
- `description`：描述
- `author` / `license` / `homepage`：发布信息
- `targets`：支持的语言和框架列表
- `dependencies`：依赖的其他规则包
- `rules`：规则文件列表

#### F1.7 Schema 校验

- Pydantic 模型验证所有字段类型和必填项
- `validate_rule()` 返回 `(Rule对象, 错误列表)`
- `validate_bundle()` 验证规则包完整性（metadata 存在、规则文件存在）
- 校验错误返回精确的字段路径（如 `conditions[0].query`）

#### F1.8 验收标准

- [ ] `Rule` / `RuleBundle` Pydantic 模型可通过 `pip install pydantic` 使用
- [ ] `validate_rule()` 可正确识别所有 Schema 违规并返回精确错误位置
- [ ] 提供至少 3 个示例规则文件（mybatis-orm、jsp-java、webxml-servlet）通过 Schema 校验
- [ ] `validate_bundle()` 可检测规则包缺失文件和格式错误

### FEATURE 2：规则引擎核心（加载器、解析器、校验器）

**需求描述**：实现规则文件的加载、解析、编译和校验全流程，为规则执行提供可用的运行时结构。

#### F2.1 规则加载器（RuleLoader）

支持从多个来源加载规则，加载顺序（后面的优先）：

1. **内置规则包**：安装路径下的 `built-in/` 目录
2. **项目本地规则**：项目根目录 `.codegraph/rules/`
3. **已安装规则包**：通过 CLI 安装的包（`cache_dir/installed/`）
4. **规则包覆盖配置**：规则配置文件（`rules.yaml`）中的 overrides

加载失败时输出警告但不阻断主流程（容错设计）。

#### F2.2 规则解析器（RuleParser）

- YAML 文件解析为 dict → Pydantic Rule 模型验证 → Rule AST 构建
- 构建时完成默认值填充和语言推断

#### F2.3 Rule AST 构建器（RuleASTBuilder）

将 Pydantic `Rule` 模型转换为运行时 `RuleAST`：

- 从 `match` 条件推断默认语言
- 解析 `from_ref` / `to_ref` 模板，提取 binding 占位符片段
- 预编译 Tree-sitter 查询（捕获语法错误）

#### F2.4 规则校验器（RuleValidator）

语义校验，在 RuleLoader 调用 Schema 校验之后运行：

- **引用完整性**：edges 中的 `{binding}` 是否在 conditions.bindings 中定义
- **binding 路径格式**：`"text"` / `"attr=X"` / `"key=X"` / `"parent.X"` 格式合法性
- **Tree-sitter 查询语法**：预编译查询是否有错误
- **循环依赖检测**：规则间的 `depends_on` 是否形成环（FEATURE5 实现）

#### F2.5 编译规则结构（CompiledRule）

运行时结构包含：

- `rule`：原始 Pydantic Rule 对象
- `ast`：RuleAST
- `source_file`：规则文件路径
- `bundle_path`：所属规则包路径
- 预编译的 Tree-sitter Query 对象

#### F2.6 验收标准

- [ ] `RuleLoader` 可从内置目录 + 项目目录 + 安装目录加载规则
- [ ] 加载失败的规则包输出警告但不阻断（容错）
- [ ] `RuleASTBuilder` 将 `Rule` Pydantic 模型完整转换为 `RuleAST`
- [ ] `RuleValidator` 检测未定义 binding 引用、查询语法错误
- [ ] 已有示例规则文件（FEATURE1 产出）全部通过加载→解析→校验全流程

### FEATURE 3：规则匹配器（Condition Evaluator）

**需求描述**：实现规则的匹配执行——从候选文件对筛选，到具体条件求值，再到绑定结果提取的完整链路。

#### F3.1 匹配过滤器（MatchFilter）

快速预过滤器，根据 `match.source` / `match.target` 的表面条件将待处理范围缩小到候选文件集：

- **Glob 文件名匹配**：使用 `fnmatch` 快速过滤
- **注解存在性检查**：通过符号表判断
- **符号 kind 过滤**：class / method / field 等
- **method_pattern 过滤**：正则表达式匹配方法名
- 返回所有满足条件的 `(source_file, target_file)` 文件对

性能要求：在 100ms 内将 10000 个文件过滤到候选集。

#### F3.2 条件执行器（ConditionEvaluator）

对单个文件对执行规则中的 conditions，提取绑定值：

- `AST_QUERY`：执行预编译的 Tree-sitter 查询，提取 capture 值
- `ANNOTATION`：从符号表注解中提取 `key=value` 对
- `NAMING`：正则表达式匹配符号名
- `REGEX`：内容全文正则匹配
- `RELATION`：符号间关系（跨文件，FEATURE5 扩展）

#### F3.3 绑定结果（BindingResult）

条件匹配成功后的绑定结果：

```python
@dataclass
class BindingResult:
    bindings: dict[str, str]   # binding_name → extracted value
    file: str                 # 来源文件
    line: int                 # 行号
    node: Tree-sitter-Node    # 可选：原始节点
    symbol: dict              # 可选：符号表条目
```

支持 `merge()` 方法合并同一文件对的多个 BindingResult。

#### F3.4 链式传播器（ChainResolver）

处理字段传播链（对应跨文件引用传播的高级场景）：

> 示例：HTTP参数 → Controller参数 → Service调用实参 → Repository方法

FEATURE3 阶段只提供骨架，链式传播逻辑在 FEATURE5 实现。

#### F3.5 验收标准

- [ ] `MatchFilter` 可在 100ms 内将 10000 个文件过滤到候选集
- [ ] `ConditionEvaluator` 正确执行 `AST_QUERY` / `ANNOTATION` / `NAMING` / `REGEX` 条件
- [ ] `BindingResult` 正确提取 `@Result(property="userId", column="user_id")` 中的属性
- [ ] 多个条件对同一文件对匹配时，结果合并正确
- [ ] `ChainResolver` 骨架就绪（链式传播逻辑在 FEATURE5 实现）

### FEATURE 4：边发射器（Edge Emitter）

**需求描述**：从匹配结果（BindingResult）生成 CodeGraph 的 Edge 对象，完成规则执行的最后一环。

#### F4.1 模板渲染器（TemplateRenderer）

渲染 `from_ref` / `to_ref` 中的 `{binding}` 占位符：

- 支持 `"{binding_name}"` 格式 → 替换为绑定变量值
- 支持 `"db::{column}"` 字面量格式
- 支持 `"meta:source"` / `"meta:target"` → 文件路径
- 尝试从符号表解析未绑定的变量名

#### F4.2 边发射器（EdgeEmitter）

根据 EmitEdge 规格从 BindingResult 生成 Edge：

- **去重**：同一 `(kind, from_qname, to_qname)` 组合只产生一条边
- **属性渲染**：properties 字典中的 `{binding}` 模板全部替换
- **行号确定**：优先使用 `line_ref` 指定的 binding 值
- **文件路径**：使用 BindingResult 的 file 字段

生成的 Edge 格式与 Python 插件完全一致，可直接合并入 `GraphContext.edges` 列表。

#### F4.3 边分类器（EdgeClassifier）

根据 `from_qname` / `to_qname` 确定边的分类：

| 分类 | 条件 |
|------|------|
| `intra_type` | 同文件类型（如同为 Java） |
| `cross_type` | 不同文件类型但同层（如同为 backend） |
| `cross_layer` | 不同层（如 frontend → backend） |

#### F4.4 验收标准

- [ ] `EdgeEmitter.emit()` 从 `BindingResult` 生成正确的 `Edge` 对象
- [ ] 模板渲染正确处理 `{binding}` 占位符替换
- [ ] 重复的 `(kind, from_qname, to_qname)` 组合只产生一条边
- [ ] 生成的 Edge 可直接合并入 `GraphContext.edges` 列表
- [ ] `EdgeEmitter.classify_edge()` 正确分类边（intra_type / cross_type / cross_layer）

### FEATURE 5：规则引擎集成（与 stage6 及现有插件协同）

**需求描述**：将规则引擎集成到 `stage6_cross_file.py`，实现与现有 Python 插件的双轨并行。

#### F5.1 整体集成架构

`stage6_cross_file.py` 的 `generate_cross_file_edges()` 函数改造为：

```
现有流程：CrossFilePlugins 生成 Cross-Type / Cross-Layer 边
    ↓
[新增] RuleEngine 加载并执行规则 → 规则引擎边
    ↓
合并 → 去重 → 分类 → 写入输出
```

规则引擎独立运行，不影响 Python 插件的执行结果。

#### F5.2 RuleEngine 主类

```python
class RuleEngine:
    def load_rules(self, enabled_ids: list[str] | None = None) -> list[CompiledRule]
    def execute(self, rules: list[CompiledRule] | None = None) -> list[Edge]
    def execute_incremental(self, changed_files: set[str], ...) -> list[Edge]
```

- `load_rules()`：加载所有可用规则，按 priority 排序
- `execute()`：执行匹配→发射全流程，返回 Edge 列表
- `execute_incremental()`：增量执行，只重跑受变更文件影响的规则

#### F5.3 便捷入口函数

```python
def run_rule_engine(
    context: GraphContext,
    project_root: Path,
    enabled_rules: list[str] | None = None,
    verbose: bool = False,
) -> list[Edge]:
```

从 `stage6_cross_file.py` 直接调用。

#### F5.4 双轨结果对比（DiffReporter）

在开发模式下，对比规则引擎和 Python 插件的输出差异：

- `rule_only`：规则引擎多产出的边
- `plugin_only`：Python 插件多产出的边
- `common`：两边共有的边
- `match_rate`：匹配率（common / max(rule_set, plugin_set)）

用于验证规则迁移的正确性。

#### F5.5 验收标准

- [ ] `RuleEngine.execute()` 在有规则的情况下正常运行，无规则时不影响现有流程
- [ ] `run_rule_engine()` 从 `stage6_cross_file.py` 调用成功
- [ ] 规则引擎产生的边与 Python 插件的边格式完全一致（可合并入同一列表）
- [ ] `DiffReporter` 可正确报告两套结果的差异（用于迁移验证）
- [ ] 启用规则引擎后，现有 `python run_pipeline.py --all` 流程不变
- [ ] `--rule-config` 可指定只启用特定规则

### FEATURE 6：规则包管理 CLI

**需求描述**：提供命令行工具，实现规则包的安装、卸载、查询、校验、测试等全生命周期管理。

#### F6.1 命令行接口

```
codegraph rule install <package-id> [version]  # 安装规则包
codegraph rule uninstall <package-id>         # 卸载规则包
codegraph rule list [--verbose]                # 列出已安装规则
codegraph rule validate <rule-file>            # 校验规则文件
codegraph rule test <rule-file> --sample <dir> # 用样本数据测试规则
codegraph rule preview <rule-file> --target <file>  # 预览规则匹配结果
```

#### F6.2 规则包安装

- 支持从 GitHub releases 安装（`codegraph rule install mybatis-community`）
- 支持从本地 tar.gz 安装（离线场景）
- 支持从指定 URL 安装（私有规则包场景）
- 安装到 `cache_dir/installed/{package-id}@{version}/`

#### F6.3 规则包管理器（BundleManager）

- `install(package_id, version)`：下载并解压规则包
- `uninstall(package_id)`：删除规则包目录
- `list_installed()`：列出所有已安装规则包及其中包含的规则

#### F6.4 run_pipeline.py 集成

新增 pipeline 参数：

- `--rule-config <path>`：规则配置文件路径
- `--with-rules <rule-id...>`：只启用指定的规则
- `--no-rules`：完全禁用规则引擎，回归纯 Python 插件模式

#### F6.5 验收标准

- [ ] `codegraph rule validate <file>` 正确校验规则文件并报告精确错误
- [ ] `codegraph rule test <rule-file> --sample <dir>` 生成边并输出前 20 条
- [ ] `codegraph rule preview <rule-file> --target <file>` 显示匹配结果和绑定值
- [ ] `codegraph rule list` 列出所有已安装规则（内置 + 项目 + 已安装）
- [ ] `--with-rules` 参数使 pipeline 只执行指定规则
- [ ] `--no-rules` 参数完全禁用规则引擎，回归纯 Python 插件模式

## 四、非功能需求

### NFR1：性能

- `MatchFilter` 在 100ms 内完成 10000 个文件的候选过滤
- 规则引擎执行时间不超过同功能 Python 插件的 1.5 倍
- 增量执行（`execute_incremental`）时间与变更文件数量线性相关

### NFR2：稳定性

- 规则文件加载失败时输出警告，不阻断主流程
- 无效规则（如 Schema 校验失败）不阻止其他规则执行
- 规则引擎内部异常（Tree-sitter 查询错误等）被捕获并报告，不导致 pipeline 崩溃

### NFR3：可维护性

- 规则文件格式对非 Python 开发者可读、可编写
- 规则包可独立版本化管理（遵循语义化版本）
- 规则元数据（name、description、tags）支持多语言

### NFR4：向后兼容

- 规则引擎不修改现有 Python 插件的行为
- 现有 `run_pipeline.py --all` 流程完全兼容（默认不启用规则引擎）
- 已有的 Python 插件继续作为 CrossFilePlugins 运行

### NFR5：可扩展性

- 新增条件类型（ConditionType）只需实现 `_eval_{type}` 方法
- 新增边类型（EdgeKind）无需修改 Python 代码，只需在 YAML 中声明
- 规则包可声明对其他规则包的依赖（`depends_on`）

## 五、范围与边界

### 5.1 明确在范围内

- 规则文件的格式设计、Schema 定义、校验逻辑
- 规则引擎的加载、匹配、发射全流程
- 规则包的 CLI 管理（安装/卸载/查询/测试）
- 与现有 `stage6_cross_file.py` 的集成
- 第一个 Python 插件（mybatis_orm）到 YAML 规则的迁移验证

### 5.2 明确不在范围内（本期）

- Python 插件的完全废弃（双轨并行永久保留）
- 规则包的远程市场/仓库（只实现本地安装，远程下载骨架为 TODO）
- 规则的可视化编辑器 Web UI（FEATURE8 只提供骨架）
- 规则的条件组合逻辑（AND/OR/NOT）的高级表达式支持
- 规则的循环依赖检测（FEATURE5 的 `depends_on` 循环检测）
- 规则的链式传播（ChainResolver 的完整实现）

### 5.3 与其他模块的边界

| 模块 | 边界 |
|------|------|
| **stage1~5** | 规则引擎复用其 Tree-sitter 解析、符号表、类型推断结果，不修改其代码 |
| **stage6_cross_file.py** | 规则引擎作为其内部的一个可选步骤，不修改 Python 插件接口 |
| **Python 插件** | 规则引擎与 Python 插件双轨并行，共享 `GraphContext`，输出格式完全一致 |
| **CLI（run_pipeline.py）** | 新增 `--rule-config` 等参数，不修改已有参数语义 |
